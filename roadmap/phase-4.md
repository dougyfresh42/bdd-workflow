## `npx bdd-workflow docs`

This command generates API documentation from JSDoc comments using TypeDoc.

### What It Does

1. Reads `typedoc.json` from the project root (the user's project, not the bdd-workflow package)
2. If `typedoc.json` doesn't exist, uses sensible defaults derived from `bdd-workflow.config.ts`
3. Runs TypeDoc programmatically (not as a shell command) using TypeDoc's Node API
4. Outputs markdown files to `config.docs.outputDir` (default: `docs/`)

### Why Programmatic (Not Shell)

Using TypeDoc's Node API instead of `npx typedoc` allows the command to:
- Apply defaults from `bdd-workflow.config.ts` without requiring a separate `typedoc.json`
- Report errors clearly rather than passing through TypeDoc's error output
- Be testable

### `src/commands/docs.ts`

```typescript
import { Command } from 'commander';
import { generateDocs } from '../generators/docs.js';
import { loadConfig } from '../config.js';

export function docsCommand(): Command {
  return new Command('docs')
    .description('Generate API documentation from JSDoc comments')
    .option('--config <path>', 'Path to bdd-workflow.config.ts')
    .option('--format <format>', 'Output format: markdown or html', 'markdown')
    .action(async (opts) => {
      const config = await loadConfig(opts.config);
      if (opts.format) config.docs.format = opts.format;
      await generateDocs(config);
      console.log(`Docs written to ${config.docs.outputDir}/`);
    });
}
```

### `src/generators/docs.ts`

```typescript
import { BddWorkflowConfig } from '../config.js';
import { Application, TSConfigReader, TypeDocReader } from 'typedoc';

export async function generateDocs(config: BddWorkflowConfig): Promise<void> {
  const app = await Application.bootstrapWithPlugins({
    entryPoints: ['src/index.ts'],
    tsconfig: 'tsconfig.json',
    out: config.docs.outputDir,
    plugin: config.docs.format === 'markdown'
      ? ['typedoc-plugin-markdown']
      : [],
    readme: 'none',
    excludePrivate: true,
    excludeInternal: true,
  }, [
    new TSConfigReader(),
    new TypeDocReader(),
  ]);

  const project = await app.convert();
  if (!project) {
    throw new Error('TypeDoc failed to parse the project');
  }

  await app.generateDocs(project, config.docs.outputDir);
}
```

## `npx bdd-workflow specs`

This command generates `SPECS.md` — a human-readable summary of all behavioral specifications derived from `.feature` files. Unlike `CONTEXT.md` (which is terse and AI-optimized), `SPECS.md` is formatted for human readers and includes more detail.

### SPECS.md Output Format

```markdown
# Behavioral Specifications

> Auto-generated by `npx bdd-workflow specs`. Do not edit manually.
> Last updated: 2026-04-02T15:30:00Z
>
> This document summarizes all behavioral specifications defined in `features/`.
> Each specification corresponds to a Gherkin `.feature` file.

---

## Authentication

**File**: `features/auth.feature`

### User Login

**Given** a registered user with valid credentials
**When** the user submits their email and password
**Then** they receive an authentication token
**And** the token is valid for 24 hours

### Login Failure

**Given** a registered user
**When** the user submits an incorrect password
**Then** they receive a 401 Unauthorized response
**And** no authentication token is issued

---

## User Profiles

**File**: `features/users.feature`

[... and so on]

---

## Summary

| Feature | Scenarios |
|---------|-----------|
| Authentication | 3 |
| User Profiles | 2 |
| **Total** | **5** |
```

### `src/commands/specs.ts`

```typescript
import { Command } from 'commander';
import { generateSpecs } from '../generators/specs.js';
import { loadConfig } from '../config.js';

export function specsCommand(): Command {
  return new Command('specs')
    .description('Generate SPECS.md from Gherkin .feature files')
    .option('--config <path>', 'Path to bdd-workflow.config.ts')
    .option('--output <path>', 'Output file path', 'SPECS.md')
    .action(async (opts) => {
      const config = await loadConfig(opts.config);
      const outputPath = opts.output;
      await generateSpecs(config, outputPath);
      console.log(`SPECS.md written to ${outputPath}`);
    });
}
```

### `src/generators/specs.ts`

Reuses the `parseFeatureFiles` parser from Phase 3, but formats for human readers with full step detail.

```typescript
import { BddWorkflowConfig } from '../config.js';
import { parseFeatureFilesDetailed } from '../parsers/gherkin.js';
import { writeFile } from 'node:fs/promises';

export interface StepDetail {
  keyword: string;  // 'Given' | 'When' | 'Then' | 'And' | 'But'
  text: string;
}

export interface ScenarioDetail {
  name: string;
  steps: StepDetail[];
  tags: string[];
  isOutline: boolean;
  examples?: string[][];  // For Scenario Outlines
}

export interface FeatureDetail {
  filePath: string;
  featureName: string;
  description: string;
  scenarios: ScenarioDetail[];
  tags: string[];
}

export async function generateSpecs(config: BddWorkflowConfig, outputPath: string): Promise<void> {
  const features = await parseFeatureFilesDetailed(config);
  const lines: string[] = [];

  lines.push('# Behavioral Specifications\n');
  lines.push(`> Auto-generated by \`npx bdd-workflow specs\`. Do not edit manually.\n> Last updated: ${new Date().toISOString()}`);
  lines.push(`>\n> This document summarizes all behavioral specifications defined in \`${config.bdd.featuresDir}/\`.\n`);
  lines.push('---\n');

  for (const feature of features) {
    lines.push(`## ${feature.featureName}\n`);
    lines.push(`**File**: \`${feature.filePath}\`\n`);

    if (feature.description) {
      lines.push(feature.description + '\n');
    }

    for (const scenario of feature.scenarios) {
      lines.push(`### ${scenario.name}\n`);

      if (scenario.tags.length > 0) {
        lines.push(`*Tags: ${scenario.tags.join(', ')}*\n`);
      }

      for (const step of scenario.steps) {
        lines.push(`**${step.keyword}** ${step.text}`);
      }
      lines.push('');
    }

    lines.push('---\n');
  }

  // Summary table
  lines.push('## Summary\n');
  lines.push('| Feature | Scenarios |');
  lines.push('|---------|-----------|');
  let total = 0;
  for (const feature of features) {
    lines.push(`| ${feature.featureName} | ${feature.scenarios.length} |`);
    total += feature.scenarios.length;
  }
  lines.push(`| **Total** | **${total}** |`);

  await writeFile(outputPath, lines.join('\n') + '\n');
}
```

## Acceptance Criteria

### `bdd-workflow docs`

- [ ] Command runs without errors in a project with TypeScript source files and JSDoc comments
- [ ] Generates markdown files in the `docs/` directory
- [ ] Generated files contain formatted API documentation from JSDoc comments
- [ ] Running with `--format html` generates HTML output instead
- [ ] Command fails gracefully if no `src/index.ts` entry point exists, with a helpful message
- [ ] TypeDoc errors are surfaced clearly (not swallowed)

### `bdd-workflow specs`

- [ ] Command runs without errors in a project with `.feature` files
- [ ] Generated `SPECS.md` contains a section for each Feature
- [ ] Each section lists all Scenarios with their full step text
- [ ] Summary table at the bottom is accurate
- [ ] Command runs gracefully when there are no `.feature` files (outputs an empty spec summary)
- [ ] Tags are included when present
- [ ] Scenario Outlines are noted as outlines (not listed as duplicate scenarios)

### Both Commands

- [ ] Both commands are registered in the CLI and visible in `--help`
- [ ] BDD scenarios exist covering the happy-path output of both commands

## Notes

- `SPECS.md` is tracked in git alongside `CONTEXT.md`. Both should be regenerated as part of the `/archive` command (update the archive command template from Phase 2 to also run `npx bdd-workflow specs`).
- The docs output in `docs/` may or may not be committed to git — this is a project-level decision. The `.gitignore` template from Phase 1 comments out the `docs/` line so users can decide.
- For Scenario Outlines, `SPECS.md` should show the outline template (with `<parameter>` placeholders), not each expanded example row — the goal is readable specifications, not test case enumeration.
- TypeDoc's markdown plugin (`typedoc-plugin-markdown`) must be installed as a dependency of `bdd-workflow` (not just a devDependency) since the docs command runs TypeDoc programmatically at the user's project level.
