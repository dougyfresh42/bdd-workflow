# Phase 3 — Context Generation

**Prerequisite reading**: [docs/design.md](design.md)
**Depends on**: Phase 1 (CLI structure), Phase 2 (bootstrap — use the workflow for this phase)

**Goal**: Implement `npx bdd-workflow context`, which generates `CONTEXT.md` — a programmatic codebase summary that gives AI agents full project understanding without manual exploration.

**This phase should be implemented using the bdd-workflow itself** (see bootstrap note in Phase 2).

---

## Why This Matters

`CONTEXT.md` is listed in `opencode.json` under `instructions`, so OpenCode automatically loads it into every session. When it's accurate, agents don't need to explore the codebase at the start of every task. This is the payoff of requiring JSDoc comments and `.feature` files for every change: the context can be derived mechanically rather than discovered by an agent.

## CONTEXT.md Output Format

The generated file must be readable by both humans and AI agents. It follows this structure:

```markdown
# Project Context

> Auto-generated by `npx bdd-workflow context`. Do not edit manually.
> Last updated: 2026-04-02T15:30:00Z

## Overview

[package.json description + @module comment from src/index.ts]

## Directory Structure

```
src/
  auth/
    session.ts
    tokens.ts
  users/
    profile.ts
features/
  auth.feature
  users.feature
```

## Modules

| File | Description |
|------|-------------|
| `src/auth/session.ts` | Manages user session lifecycle, including creation, validation, and expiry. |
| `src/auth/tokens.ts` | JWT token generation and verification for stateless authentication. |

## Features

### Authentication (`features/auth.feature`)
- Scenario: User logs in with valid credentials
- Scenario: User is rejected with invalid password
- Scenario: Session expires after configured timeout

### User Profiles (`features/users.feature`)
- Scenario: User retrieves their profile
- Scenario: User updates their display name

## Public API

### `src/auth/session.ts`
- `createSession(userId: string, opts?: SessionOpts): Promise<Session>`
- `validateSession(token: string): Promise<SessionInfo | null>`
- `expireSession(token: string): Promise<void>`
```

## Implementation

### `src/commands/context.ts`

```typescript
import { Command } from 'commander';
import { generateContext } from '../generators/context.js';
import { loadConfig } from '../config.js';

export function contextCommand(): Command {
  return new Command('context')
    .description('Regenerate CONTEXT.md from source files and feature specs')
    .option('--config <path>', 'Path to bdd-workflow.config.ts')
    .action(async (opts) => {
      const config = await loadConfig(opts.config);
      await generateContext(config);
      console.log(`CONTEXT.md updated.`);
    });
}
```

### `src/generators/context.ts`

Assembles the sections in order:

```typescript
import { BddWorkflowConfig } from '../config.js';
import { generateDirectoryTree } from './structure.js';
import { parseFeatureFiles } from '../parsers/gherkin.js';
import { extractModuleSummaries } from '../parsers/jsdoc.js';
import { extractPublicApi } from '../parsers/typescript.js';
import { readFile, writeFile } from 'node:fs/promises';
import { join } from 'node:path';

export async function generateContext(config: BddWorkflowConfig): Promise<void> {
  const sections: string[] = [];

  sections.push(`# Project Context\n`);
  sections.push(`> Auto-generated by \`npx bdd-workflow context\`. Do not edit manually.\n> Last updated: ${new Date().toISOString()}\n`);

  // Section: Overview
  if (config.context.sections.structure) {
    const overview = await buildOverview(config);
    sections.push(`## Overview\n\n${overview}`);
  }

  // Section: Directory Structure
  if (config.context.sections.structure) {
    const tree = await generateDirectoryTree(config);
    sections.push(`## Directory Structure\n\n\`\`\`\n${tree}\`\`\``);
  }

  // Section: Module Summaries
  if (config.context.sections.moduleSummaries) {
    const modules = await extractModuleSummaries(config);
    sections.push(formatModulesSection(modules));
  }

  // Section: Feature Summaries
  if (config.context.sections.featureSummaries) {
    const features = await parseFeatureFiles(config);
    sections.push(formatFeaturesSection(features));
  }

  // Section: Public API
  if (config.context.sections.exports) {
    const api = await extractPublicApi(config);
    sections.push(formatApiSection(api));
  }

  await writeFile(config.context.outputFile, sections.join('\n\n') + '\n');
}
```

### `src/parsers/gherkin.ts`

Parses `.feature` files using `@cucumber/gherkin` and returns a structured summary.

**Dependencies**: `@cucumber/gherkin`, `@cucumber/messages`

```typescript
import * as Gherkin from '@cucumber/gherkin';
import { IdGenerator } from '@cucumber/messages';

export interface FeatureSummary {
  filePath: string;
  featureName: string;
  scenarios: string[];
}

export async function parseFeatureFiles(config: BddWorkflowConfig): Promise<FeatureSummary[]> {
  // 1. Glob for all .feature files in config.bdd.featuresDir
  // 2. For each file, use @cucumber/gherkin Parser to build an AST
  // 3. Extract: Feature name, all Scenario names (including Scenario Outline names)
  // 4. Return array of FeatureSummary
}
```

**Key `@cucumber/gherkin` usage**:

```typescript
const uuidFn = IdGenerator.uuid();
const builder = new Gherkin.AstBuilder(uuidFn);
const matcher = new Gherkin.GherkinClassicTokenMatcher();
const parser = new Gherkin.Parser(builder, matcher);

const content = await readFile(featurePath, 'utf-8');
const gherkinDocument = parser.parse(content);

const featureName = gherkinDocument.feature?.name ?? '(unnamed)';
const scenarios = gherkinDocument.feature?.children
  ?.map(child => child.scenario?.name ?? child.rule?.name)
  .filter(Boolean) ?? [];
```

### `src/parsers/jsdoc.ts`

Extracts file-level JSDoc comments (the `@module` / first block comment) from TypeScript source files.

**Approach**: Use the TypeScript compiler API to parse source files and extract the leading JSDoc comment of each file. Do not use regex — the TS compiler handles edge cases correctly.

```typescript
import ts from 'typescript';

export interface ModuleSummary {
  filePath: string;
  description: string;
}

export async function extractModuleSummaries(config: BddWorkflowConfig): Promise<ModuleSummary[]> {
  // 1. Glob for all .ts files in config.context.include
  // 2. For each file, create a TS SourceFile (ts.createSourceFile)
  // 3. Get the first JSDoc comment (ts.getJSDocTags or leading trivia)
  // 4. Extract the description text
  // 5. Return array of ModuleSummary, skipping files with no file-level comment
}
```

**Extracting the file-level comment**:

```typescript
function getFileComment(sourceFile: ts.SourceFile): string | null {
  const firstStatement = sourceFile.statements[0];
  if (!firstStatement) return null;

  const ranges = ts.getLeadingCommentRanges(sourceFile.text, firstStatement.pos);
  if (!ranges || ranges.length === 0) return null;

  // Find the JSDoc block (/** ... */)
  for (const range of ranges) {
    const text = sourceFile.text.slice(range.pos, range.end);
    if (text.startsWith('/**')) {
      return parseJsDocText(text);
    }
  }
  return null;
}
```

### `src/parsers/typescript.ts`

Extracts exported function and class signatures from TypeScript source files.

**Approach**: Use the TypeScript compiler API. Create a Program from the project's `tsconfig.json`, then walk the type checker to get exported symbols and their signatures.

```typescript
import ts from 'typescript';

export interface ApiEntry {
  filePath: string;
  signatures: string[];
}

export async function extractPublicApi(config: BddWorkflowConfig): Promise<ApiEntry[]> {
  // 1. Load tsconfig.json from cwd
  // 2. Create a ts.Program
  // 3. For each source file (matching include patterns):
  //    a. Get the module symbol
  //    b. Get exported declarations
  //    c. Format each as a human-readable signature string
  // 4. Return array of ApiEntry
}
```

**Key technique — formatting signatures**:

```typescript
function formatSignature(checker: ts.TypeChecker, symbol: ts.Symbol): string {
  const decl = symbol.declarations?.[0];
  if (!decl) return symbol.name;

  if (ts.isFunctionDeclaration(decl) || ts.isMethodDeclaration(decl)) {
    const sig = checker.getSignatureFromDeclaration(decl);
    if (sig) return checker.signatureToString(sig);
  }

  if (ts.isClassDeclaration(decl)) {
    return `class ${symbol.name}`;
  }

  const type = checker.getTypeOfSymbolAtLocation(symbol, decl);
  return `${symbol.name}: ${checker.typeToString(type)}`;
}
```

## `loadConfig` Implementation (update `src/config.ts`)

The config loader needs to import `bdd-workflow.config.ts` from the project being processed (not the bdd-workflow package itself):

```typescript
export async function loadConfig(configPath?: string): Promise<BddWorkflowConfig> {
  const resolved = configPath
    ? resolve(configPath)
    : join(process.cwd(), 'bdd-workflow.config.ts');

  if (!existsSync(resolved)) {
    console.warn('No bdd-workflow.config.ts found, using defaults.');
    return defineConfig({});
  }

  // Use tsx or ts-node to import the config
  // The config exports a default BddWorkflowConfig
  const mod = await import(resolved);
  return mod.default ?? defineConfig({});
}
```

Note: Dynamic import of `.ts` files requires the CLI to run under `tsx` or `ts-node`. Configure `package.json`:

```json
{
  "bin": {
    "bdd-workflow": "./dist/cli.js"
  }
}
```

For the dist build, config loading should compile to a version that uses `tsx` as a child process to evaluate the config file, since the config file lives in user projects (not pre-compiled).

## Notes

- The TypeScript compiler API (`typescript` package) is used for both JSDoc extraction and signature extraction. This avoids regex-based parsing and handles edge cases correctly (generics, overloads, decorators).
- The `@cucumber/gherkin` parser is the official Gherkin parser — it handles all Gherkin syntax including Rules, Scenario Outlines, and Examples tables.
- `CONTEXT.md` should be committed to git. The `/archive` command regenerates it after each change.
- If a source file has no file-level JSDoc comment, it is silently skipped in the modules section. The workflow's review step will catch this omission on the next change to that file.
